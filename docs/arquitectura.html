Arquitectura & despliegue

Cómo conectar PRAIA Desktop a tu data real y compartirla con tu equipo.

Dos caminos según prisa y madurez. Mismo prototipo, distinto motor por debajo. Te explico qué cambia, cómo se loguea la gente, dónde ves quién hizo qué — y cómo lo pones en producción esta semana.

01 — Las 3 piezas

Una app web siempre se compone de tres cosas.

Hoy AppSheet te las da todas en una sola caja negra. Cuando migras a una app propia las separas — eso es lo que da el control que la app actual no tiene.

PIEZA 1

Datos

Dónde vive realmente la información: tickets, apartamentos, agentes. Hoy: Google Sheet. Mañana: Sheet o base de datos.

PIEZA 2

Autenticación

Quién puede entrar y qué puede ver/hacer. Login, sesiones, roles, MFA, recuperar contraseña.

PIEZA 3

App (UI)

Lo que ves hoy en PRAIA Desktop.html. Vive en un servidor de hosting estático; cualquiera con el link la ve.

02 — Dos caminos

Rápido y barato, o robusto y escalable.

Ambos llevan a una app web real que tu equipo usa todos los días. El A se monta en 1–2 días si eres ágil. El B en 1–2 semanas y aguanta crecimiento.

CAMINO A · rápido

Google Sheets + Apps Script

Mantienes la Sheet "Tickets" como base de datos. Apps Script la expone como API. La app es la que ya viste.

● Google Sheet ● Apps Script ● Google OAuth ● Netlify / Vercel
  • Listo en 1–2 días
  • Cero costo · cuota Google ya incluida
  • Mantén tu workflow actual con Sheets
  • Login con cuenta Google de tu empresa
  • Lento con >5 000 filas o >20 usuarios simultáneos
  • Auditoría limitada (depende de "Versiones" de Sheets)
  • Apps Script tiene cuotas duras (6 min/ejecución)
tiempo · 1–2 días costo · $0/mes
CAMINO B · profesional · recomendado

Supabase + Vercel

Migras la Sheet a una base Postgres con auth, audit, realtime y API gratis. Tu data crece sin frenos.

● Supabase (Postgres) ● Supabase Auth ● Vercel ● Row-level security
  • Auditoría completa por fila (quién, qué, cuándo)
  • Roles y permisos finos (RLS de Postgres)
  • Realtime: el dashboard se actualiza solo
  • Escala a millones de filas sin problema
  • Backups automáticos
  • Requiere migrar la Sheet → tabla Postgres (1 vez)
  • Curva inicial mayor
tiempo · 1–2 semanas costo · $0–25/mes
Mi recomendación: empieza por el Camino A esta semana para tener al equipo usándolo ya, y migra a Supabase en 1–2 meses cuando sepas qué funciona y qué hay que ajustar. El frontend (la app que ya viste) no cambia entre los dos caminos — solo cambia qué llama por detrás.
03 — Camino A · rápido

Google Sheets como base de datos, Apps Script como API.

Tu Sheet "Tickets" sigue siendo la fuente de verdad. Apps Script publica un endpoint que la app consulta. Todo se queda en tu workspace de Google.

Datos
Google Sheet Hojas: Tickets · Apartamentos · Agentes · Audit_Log sheets.google.com
→REST
API
Apps Script Web App que expone /getTickets, /createTicket, etc. script.google.com
→fetch
App
PRAIA Desktop El HTML que ya viste, alojado en hosting estático con dominio propio. praia.tudominio.com

Cómo se ve un endpoint en Apps Script

// En tu Sheet → Extensions → Apps Script
function doGet(e) {
  const sheet = SpreadsheetApp.getActive().getSheetByName('Tickets');
  const rows = sheet.getDataRange().getValues();
  const headers = rows.shift();
  const tickets = rows.map(r => Object.fromEntries(headers.map((h,i) => [h, r[i]])));
  return ContentService
    .createTextOutput(JSON.stringify(tickets))
    .setMimeType(ContentService.MimeType.JSON);
}

Lo publicas con Deploy → New deployment → Web app, eliges "ejecutar como yo" + "acceso: dominio de mi organización" y obtienes una URL. La app le pega ahí. Listo.

04 — Camino B · profesional

Supabase como cerebro de PRAIA.

Supabase te da Postgres + Auth + API + audit + realtime en un solo dashboard. Es lo que usan startups que arrancan en serio.

Datos + Auth
Supabase Postgres con tablas: tickets, apartments, agents, audit_log, users. supabase.com
→REST + WS
Cliente JS
Supabase SDK Llamadas directas seguras desde el browser. RLS protege qué ve cada usuario. @supabase/supabase-js
→
App
PRAIA Desktop El mismo HTML, alojado en Vercel. praia.tudominio.com

Modelo de datos sugerido

-- Tablas mínimas
create table users     (id uuid pk, email, name, role, shift, created_at);
create table buildings (id text pk, name, address, units int);
create table apartments(id text pk, building_id ref buildings, beds, baths, capacity);
create table tickets   (id uuid pk, status, priority, platform, issue,
                         apartment_id ref apartments, guest_name,
                         agent_id ref users, description,
                         created_at, updated_at, closed_at, sla_deadline);
create table comments  (id uuid pk, ticket_id ref tickets, author_id ref users, body, created_at);
create table audit_log (id, table_name, row_id, action, actor_id, before, after, ts);

Lo que ganas inmediatamente

  • Realtime. Cuando Luis cierra un ticket, María lo ve actualizado al instante sin recargar.
  • RLS (Row-Level Security). Un agente de "Limpieza" no ve tickets de "Mantenimiento" — la base misma lo bloquea.
  • Audit log automático. Un trigger de Postgres logea cada INSERT/UPDATE/DELETE.
  • API REST gratis. Cualquier sistema externo (Guesty, Zapier, etc.) puede consultar tickets vía API.
05 — Login

Así entra tu equipo a PRAIA.

Opción simple: cuentas Google ya existentes (Workspace). Opción flexible: email + contraseña o magic link sin tener que crear cuentas Google.

PASO 01
Visitan praia.tudominio.com

Pantalla de login si no hay sesión.

PASO 02
"Entrar con Google"

O email + contraseña en Camino B.

PASO 03
Validación

El backend chequea email contra la tabla users.

PASO 04
Asigna rol

Supervisor, Operaciones, Limpieza, Mantenimiento.

PASO 05
Entra a la app

Ve únicamente lo que su rol permite.

Onboarding de un agente nuevo

  1. Vas a Ajustes → Equipo → Invitar.
  2. Pones email + rol + turno. Sistema le manda magic link.
  3. Clic, queda dentro. Sin contraseñas que recordar.
Importante: el email del usuario es la clave de identidad en ambos caminos. Cada acción quedará registrada con ese email — eso es la base de la trazabilidad.

07 — Trazabilidad

Dónde ves quién hizo qué y cuándo.

Cada acción importante deja huella. Lo encuentras en una vista Auditoría dentro de Ajustes — filtrable por usuario, fecha, tipo de acción, y entidad afectada (ticket, apartamento, usuario).

Audit log · últimas 8 entradas tiempo real

24-05 18:30 Luis M. CREATE TK-1042 · apto 7834-3 · issue Plomería
24-05 18:25 Carla O. UPDATE TK-1031 · agente: Luis → José
24-05 18:21 José C. UPDATE TK-1029 · estado: proceso → cerrado
24-05 18:14 María P. DELETE TK-1018 · razón: duplicado de TK-1015
24-05 18:05 Diego S. LOGIN sesión iniciada · IP 190.92.x.x · Mac · Chrome
24-05 17:58 Ana R. UPDATE TK-1012 · prioridad: media → alta
24-05 17:42 María P. CREATE USER Diego Soto · rol Mantenimiento · turno Tarde

Cómo se logea esto

  • Camino A: cada llamada de la app a Apps Script escribe una fila en una hoja Audit_Log de la misma Google Sheet. Además Sheets ya tiene historial nativo (Archivo → Historial de versiones).
  • Camino B: un trigger de Postgres en cada tabla escribe automáticamente a audit_log en cada INSERT/UPDATE/DELETE, con campos before y after en JSON. Súper consultable.
08 — Compartir

Cómo se lo entregas al equipo.

El HTML que ya tienes se sube a un hosting estático y queda detrás de tu dominio. Cada miembro entra desde su navegador — mismo link, distinta vista según su rol.

Comprar dominio (opcional)
Algo como praia.tudominio.com o usa subdominio del que ya tengas. Si no quieres dominio, igual queda en praia-app.vercel.app gratis.
Cuenta gratis en Vercel o Netlify
Subes el HTML. Listo. Cada cambio que hagas se redespliega solo si lo conectas a GitHub.
Conectar backend
Pones la URL de Apps Script (Camino A) o las keys de Supabase (Camino B) en variables de entorno del hosting.
Invitar al equipo
Vas a Ajustes → Equipo dentro de PRAIA. Agregas email + rol + turno. Reciben un magic link y entran.
Apps móviles "instalables"
Convirtiéndola en PWA (1 archivo extra) el equipo puede "instalar" PRAIA en el celular como si fuera una app nativa — abre offline y todo. Sin App Store.

10 — Plan recomendado

Mis siguientes 14 días si fuera tú.

Día 1–2 · Validamos el prototipo con 2 agentes
Les muestras la app actual (mock data) y anotas qué les sobra, qué les falta, qué columnas necesitan que hoy no veo.
Día 3 · Finalizar modelo de datos
Cierras qué columnas tendrán tickets, apartments, users, audit_log.
Día 4–6 · Camino A en producción
Apps Script + tu Sheet "Tickets" + Vercel. Login con Google de tu Workspace. Equipo empieza a usarlo en paralelo a AppSheet.
Día 7–10 · Audit + roles finos
Agregar la hoja Audit_Log y matriz de roles. Vista de Auditoría dentro de Ajustes.
Día 11–14 · Decisión Camino B
Si ya hay tracción y el equipo está cómodo: migración planificada a Supabase con backup de la Sheet como respaldo histórico.
